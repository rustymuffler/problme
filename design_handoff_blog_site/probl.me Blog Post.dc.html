How I run a 12-agent content pipeline

Richard Muffler | Jun 21, 2026 | 8 min read

For the last two months, almost everything published on this blog was drafted, researched, reviewed, and shipped by a small team of Claude agents working on a schedule. I show up in exactly two places: a short interview at the start, and the merge button at the end. Here is how the whole thing is wired.

I want to be honest up front. The first version broke in a way I did not expect, and the fix changed how I think about reviewing AI work. More on that near the end.

The shape of the pipeline

Every article moves through the same sequence of agents. Each one has a single job, and no agent reviews its own output. That last rule matters more than any other, so I made it structural rather than a guideline.

The handoffs run as a loop. When one agent finishes, the next is spawned automatically with the previous agent’s output as its input. The only manual steps are mine.

pipeline.txt copy 
# content loop
Interview  → Research  → Writer
           → Image  → SEO  → Proofread
           → Publisher (opens draft PR)

If the SEO reviewer or the proofreader sends feedback, the writer revises and the loop continues forward. I capped it at three revision passes before it escalates to me, so a disagreement between two agents can never spin forever.

Why no agent reviews itself

The temptation is to let the writer agent also check its own work. It is faster and cheaper. It is also useless. An agent grading its own output tends to confirm what it already decided. So the reviewer is always a separate agent with a separate checklist.

The writer that drafts an article cannot be the agent that approves it. Make that a wall, not a suggestion.

This is the same instinct from code review, just applied to prose. The proofreader scans for tone, banned phrases, and punctuation. The SEO reviewer checks structure and metadata. Neither one wrote the draft, so neither one is defending it.

Where it broke first

The first real run produced a clean, well-structured article that said almost nothing only I could say. It was technically correct and completely generic. The interview step existed, but the writer treated it as optional background instead of the spine of the piece.

The fix was small and structural: the interview notes are now mandatory source material, and the proofreader explicitly checks that my own perspective is present and identifiable. One line in a checklist changed the output more than any prompt tuning did.

If you are building something similar, that is the thing I would tell you that most write-ups skip. The interesting failures are not model failures. They are process gaps, and you fix them in the workflow, not the prompt.
